Deploy

Fly.io — CI/CD automático via GitHub Actions a cada push para main.

✅

O deploy acontece automaticamente quando o CI passa. Siga este checklist apenas para o primeiro deploy.

Pré-requisitos

  • Conta no Fly.io
  • flyctl instalado:
bash 
# Windows
winget install Fly.io.flyctl

# macOS / Linux
curl -L https://fly.io/install.sh | sh

Checklist de primeiro deploy

1. Autenticar no Fly.io

bash 
flyctl auth login

Abre o browser para login. Necessário apenas uma vez por máquina.

2. Criar o app e o volume

bash 
flyctl apps create feitu
flyctl volumes create feitu_data --region gru --size 1

O banco H2 é armazenado no volume em /data/feitu.mv.db. O volume sobrevive a deploys e restarts.

3. Gerar chaves VAPID

bash 
npx web-push generate-vapid-keys

Saída: Public Key: BAxxx... e Private Key: yyy.... Guarde os dois valores.

4. Configurar secrets no Fly.io

bash 
flyctl secrets set \
  JWT_SECRET="sua-string-aleatoria-com-minimo-32-chars" \
  VAPID_PUBLIC_KEY="coloque-a-public-key-aqui" \
  VAPID_PRIVATE_KEY="coloque-a-private-key-aqui" \
  DB_PATH="./data/feitu"

DB_PATH define o caminho do arquivo H2 sem extensão (padrão: ./data/feitu → cria /data/feitu.mv.db no volume). Para resetar o banco em produção, aponte para um novo caminho: flyctl secrets set DB_PATH=./data/feitu2.

Para gerar o JWT_SECRET no PowerShell:

PowerShell 
[System.Convert]::ToBase64String((1..32 | ForEach-Object { Get-Random -Max 256 }))
💡

Se não quiser notificações push, defina VAPID_PUBLIC_KEY e VAPID_PRIVATE_KEY como string vazia — o app desabilita push silenciosamente.

5. Configurar o token no GitHub

bash 
flyctl tokens create deploy -x 999999h

Copie o valor e adicione ao GitHub:

  1. Acesse o repositório no GitHub
  2. Settings → Secrets and variables → Actions
  3. New repository secret → Name: FLY_API_TOKEN → Value: (cole o token)

6. Primeiro push — dispara o deploy

bash 
git push origin main

O workflow deploy.yml é acionado automaticamente. Acompanhe em GitHub → Actions → Deploy ou flyctl logs. O workflow faz:

  • 1Build do frontend (npm ci && npm run build)
  • 2Build do .jar com frontend embutido (mvn package -DskipTests)
  • 3flyctl deploy --remote-only — Fly.io constrói a imagem Docker remotamente

7. Verificar que está no ar

bash 
curl https://feitu.fly.dev/actuator/health
# Esperado: {"status":"UP"}

flyctl open

Deploys subsequentes

Qualquer git push origin main dispara CI e, se passar, o deploy automático.

Para deploy manual:

bash 
mvn package -DskipTests
flyctl deploy

Configuração do Fly.io

app = "feitu"
primary_region = "gru"   # São Paulo

[[mounts]]
  source      = "feitu_data"
  destination = "/data"   # onde o H2 persiste os dados

[[vm]]
  memory   = "256mb"
  cpu_kind = "shared"
  cpus     = 1

O container expõe a porta 8080. O Fly.io redireciona 80→443 (HTTPS) automaticamente. Health check em GET /actuator/health a cada 30 segundos.

Limites de memória JVM

O Dockerfile define:

JAVA_OPTS="-Xmx128m -Xms64m -XX:+UseContainerSupport"

Com 256 MB de RAM no Fly.io, isso deixa ~128 MB para o sistema operacional e overhead do container. Se necessário, ajuste memory no fly.toml e -Xmx no Dockerfile.

Logs e debugging em produção

bash 
# Logs em tempo real
flyctl logs

# Habilitar console H2 temporariamente (use com cuidado)
flyctl secrets set H2_CONSOLE_ENABLED=true
# Acesse: https://feitu.fly.dev/h2-console
# JDBC URL: jdbc:h2:file:/data/feitu
# Desabilitar após uso:
flyctl secrets unset H2_CONSOLE_ENABLED

# SSH no container
flyctl ssh console
