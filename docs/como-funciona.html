Como funciona

Workspaces, tipos de tarefa, conclusão por checkbox, analytics e notificações push.

Workspaces

Workspaces são abas de organização. Cada workspace tem um nome e uma cor, e contém suas próprias tarefas. O limite é 3 workspaces por usuário — criação e deleção estão em Configurações (/settings). A sugestão é separar por contexto: Trabalho, Estudos, Projetos Pessoais.

Tipos de tarefa

Cada tarefa tem um tipo que define quando ela aparece:

TipoDescriçãoCampo de data
PONTUAL Aparece somente na data criada (ex: "Preparar apresentação hoje") data: LocalDate
RECORRENTE Aparece nos dias da semana configurados, toda semana (ex: "Stand-up SEG,QUA,SEX") diasSemana: "SEG,QUA,SEX"

Tarefas recorrentes são criadas uma única vez e aparecem automaticamente nos dias corretos. A conclusão de uma recorrente em um dia específico é registrada em ConclusaoRecorrente — o checkbox reseta à meia-noite (UTC-3).

Como a conclusão funciona

Ao marcar um checkbox, o frontend chama POST /api/tarefas/{id}/toggle?date=YYYY-MM-DD:

  • PONTUAL: alterna o campo concluida (boolean) na entidade Tarefa
  • RECORRENTE: cria ou remove um registro em ConclusaoRecorrente(tarefa_id, data) com constraint UNIQUE

A API retorna o estado atualizado da tarefa com concluida: true/false. O frontend atualiza a store Pinia localmente sem recarregar a lista.

Resumo diário (Analytics)

Ao acessar a tela de Resumo, o frontend consulta GET /api/analytics/daily?date=YYYY-MM-DD. O backend:

  1. Busca todos os workspaces do usuário
  2. Para cada workspace, carrega as tarefas do dia (pontuais da data + recorrentes do dia da semana)
  3. Conta total de tarefas e quantas estão concluídas
  4. Calcula percentualConcluido e flag diaCompleto

O frontend exibe três estados: A (parcial — barras de progresso por workspace), B (tudo feito — animação de celebração), C (sem tarefas no dia).

Notificações push

O Feitu suporta Web Push Notifications via VAPID. O fluxo completo:

  1. Usuário abre o app → browser pede permissão de notificação
  2. Se aceito → frontend gera uma PushSubscription via Service Worker
  3. Frontend envia endpoint + chaves (p256dh, auth) para POST /api/push/subscribe
  4. Backend persiste em PushSubscription associado ao usuário
  5. Todo minuto, o scheduler verifica se algum usuário tem horario_notificacao == agora
  6. Se sim, e se ainda não enviou hoje → envia a notificação push
  7. A notificação aparece mesmo com o app fechado
💡

O horário padrão é 22:00 e pode ser alterado em configurações via PUT /api/usuarios/me/notificacao. Se as chaves VAPID não estiverem configuradas no servidor, o push é desabilitado silenciosamente — o resto do app funciona normalmente.

PWA (Progressive Web App)

O frontend é configurado como PWA via vite-plugin-pwa. Isso permite:

  • Instalar o app no celular/desktop como se fosse nativo (ícone na tela inicial)
  • Funcionar offline para visualização — as páginas já visitadas ficam em cache via Service Worker
  • Receber notificações push mesmo com o app fechado, via Service Worker

O Service Worker usa a estratégia NetworkFirst para todas as chamadas /api/ — tenta o servidor primeiro, cai para o cache se offline.

Ações da tarefa (⋮)

Cada tarefa tem um botão ⋮ na extremidade direita da linha. Ao tocar:

  1. Um popover aparece com duas opções: "Ver descrição" e "Excluir tarefa"
  2. "Ver descrição" abre um bottom sheet com o nome, tipo/horário, campo de descrição (visualização ou edição) e botão de excluir
  3. No modo edição, a descrição é salva via PATCH /api/tarefas/{id}/descricao
  4. "Excluir tarefa" chama DELETE /api/tarefas/{id} e remove da lista

Invite code

O cadastro exige um código de convite válido (formato FEITU-XXXX). Os códigos são de uso único — cada código é consumido no momento do cadastro. Administradores inserem códigos diretamente via SQL no banco ou console H2.
